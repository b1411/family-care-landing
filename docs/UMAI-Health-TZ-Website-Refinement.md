# ТЗ на доработку сайта UMAI Health до финального состояния

**Документ:** Техническое задание для команды (фронтенд-разработчик, UI-дизайнер, копирайтер)
**Объект:** www.umai-health.kz (главная, /for-clinics, /for-families, /demo)
**Цель:** Привести работающий сайт к профессиональному уровню без изменения продуктового обещания и коммерческой модели

---

## 0. О документе

### 0.1. Принципы работы

1. **Ничего не ломать.** Все страницы остаются на своих URL. Структура навигации сохраняется.
2. **Демо-режим обязателен.** Функция «попробовать демо в 4 ролях» остаётся. Допустимы улучшения UX демо — не удаление.
3. **Форма заявки не расширяется.** Трогаем только вёрстку и валидацию, поля остаются теми же.
4. **Цены, тарифы, стоимость — нигде не указываются.** Ни в прямой форме, ни в виде «от ₸X», ни в виде ROI в тенге. Коммерческая модель остаётся в формате «обсудить подключение».
5. **Все числа на сайте имеют реальное обоснование.** Операционные метрики (события, прививки, напоминания, часы экономии) — считаем честно. Нулевые счётчики типа «0+» удаляются или заменяются на расчётные значения.
6. **White-label обещание сохраняется.** Всё, что сказано про бренд клиники, работает как есть.

### 0.2. Роли и ответственные

| Трек | Ответственный |
|---|---|
| Фронтенд и компоненты | Frontend-разработчик |
| Типографика, отступы, палитра, иконки | UI-дизайнер |
| Тексты и переформулировки | Копирайтер |
| Демо-сценарии | Продуктовый дизайнер + фронтенд |
| SEO, производительность, accessibility | Фронтенд |

### 0.3. Общий срок

При одном фронтенд-разработчике, полставки дизайнера и копирайтере на фазе 2 — **~4 недели** суммарно, при параллельных треках.

## Фаза 0 — Критические баги (1–2 дня, исправить до любых других правок)

### 0.1. Сломанные счётчики «0+» на главной

**Проблема.** В блоке метрик под Hero-секцией главной страницы показываются нули:
- «0+ Недель маршрута беременности»
- «0+ Прививок в календаре»
- «0 Дней сопровождения в первый год»
- «0+ Точек контроля в маршруте»

**Причина.** CountUp-анимация инициализируется со значения `0` и обновляется только когда блок попадает в viewport через IntersectionObserver. До этого момента (и в случае SSR, превью для соцсетей, медленной загрузки) — видны нули.

**Решение.**

1. Начальное значение счётчика = финальному значению. Рендерится сразу в HTML на сервере.
2. Анимация (если сохраняется) — опциональная надстройка. При запуске скроллит значение от 80% финального к 100%, не от нуля.
3. Все значения жёстко прописаны как константы в коде, не подтягиваются из API, не зависят от клиентской инициализации.

**Финальные значения (подставить вместо нулей):**

| Метрика | Значение | Обоснование |
|---|---|---|
| Маршрут беременности | 40 недель | Стандарт акушерства |
| Прививок в нацкалендаре РК | 18+ | Нацкалендарь первых 24 месяцев |
| Дней сопровождения | 730 | 2 года от зачатия до 2 лет ребёнка |
| Точек контроля в маршруте | 50+ | 3 УЗИ + ~15 анализов + ~10 визитов (беременность) + 5 событий (роды и первые дни) + ~10 прививок и 4 осмотра (1-й год) + 2 ревакцинации и 4 осмотра (2-й год) = ~56 |

### 0.2. Счётчик «Упущенная выручка: 0 ₸/год» на /for-clinics

**Проблема.** Показывается ноль. Также по политике проекта деньги на сайте не указываются.

**Решение.** Блок полностью заменить на операционную метрику без денежного эквивалента.

**Новая формулировка:**

> **Пропущенных точек контроля в год:** ~1 200+
>
> *При 30 новых семьях в месяц и типичной воронке «100 → 5». Каждая выпавшая семья теряет десятки запланированных событий маршрута — анализов, прививок, осмотров.*

**Расчёт (для референсов команды, на сайте не показывать):**

- 30 семей × 12 мес = 360 семей/год
- Воронка потерь ~95% за 2 года → выпадает ~340 семей
- Каждая выпавшая семья недополучает в среднем ~4 события в оставшийся период = ~1 360 событий
- Округлённая консервативная оценка для сайта: «1 200+»

### 0.3. Deeplink демо-роли

**Проблема.** Кнопка на главной «Попробовать эту роль в демо» ведёт на `/demo?role=mom`. По факту страница с этим параметром показывает общий селектор ролей, а не кабинет мамы.

**Решение — вариант А (предпочтительный).** Реализовать реальный deeplink:

- `/demo?role=mom` → автоматически открывает экран мамы (эквивалент клика по «Войти как мама»)
- Поддерживаемые значения параметра: `mom`, `coordinator`, `doctor`, `director`
- Невалидные значения → нормальный селектор + лог в консоль
- На странице демо при deeplink — хлебные крошки: «Все роли › Мама», с возможностью вернуться к селектору

**Решение — вариант Б (если A занимает больше 1 дня).** На главной CTA «Попробовать эту роль в демо» меняем на прямую ссылку сразу в кабинет мамы, минуя селектор, в новой вкладке.

### 0.4. Слипшиеся тексты в карточках — самый массовый баг

**Проблема.** В карточках, таймлайнах, мокапах телефона текстовые блоки идут слитно, без отступа. Примеры по сайту:

- «Витамин D3Ежедневно, 08:00Принять»
- «Педиатр — осмотрЗавтра, 10:30»
- «Др. Алия К.Педиатр · завтра»
- «Анализ кровиРезультат готов»
- «АКДС — 1-я дозаЧерез 5 дней»
- «Клиника АЗаписаться»
- «100 семей55 семей33 семьи15 семей5 семей»
- «НапоминаниеВитамин D3 — 08:00Принять»

**Причина.** В компонентах карточек отсутствует единая система внутренних отступов (gap во flex/grid-контейнерах). Элементы title / meta / description / action прилеплены друг к другу.

**Решение.** Ввести жёсткий внутренний ритм карточек через CSS-переменные:

```css
--card-gap-xxs: 4px;   /* внутри одной строки (имя → роль через · ) */
--card-gap-xs: 8px;    /* title → badge */
--card-gap-sm: 12px;   /* title → meta */
--card-gap-md: 16px;   /* title-block → description */
--card-gap-lg: 24px;   /* section → action */
```

**Применить ко всем компонентам:**

- `<TaskCard>` — задача в очереди координатора
- `<TimelineItem>` — шаг маршрута, событие в календаре
- `<ReminderRow>` — строка напоминания «Витамин D3 — 08:00 — Принять»
- `<AppointmentCard>` — карточка приёма
- `<FamilyRow>` — строка семьи в списке
- `<VaccineRow>` — прививка в календаре
- `<ModuleCard>` — карточка модуля на главной
- Мокапы телефона на всех трёх страницах

**Критерий приёмки.** После правки в DevTools не должно быть ни одной пары соседних текстовых элементов внутри карточки с фактическим отступом 0 px. QA проходит по всем карточкам в Chrome Inspector, проверяет computed style `gap` и `margin`.

## Фаза 1 — Типографика и отступы (1 неделя, ключевая фаза)

### 1.1. Единая типографическая шкала

Ввести как дизайн-токены. Применить на всех страницах.

| Уровень | Desktop | Mobile | Начертание | Line-height |
|---|---|---|---|---|
| Display / Hero H1 | 64 px | 40 px | 600 / 700 | 1.05 |
| H1 страницы | 48 px | 32 px | 600 | 1.1 |
| H2 секции | 36 px | 28 px | 600 | 1.15 |
| H3 подсекции | 24 px | 20 px | 600 | 1.2 |
| H4 карточки / FAQ | 18 px | 17 px | 600 | 1.3 |
| Body large | 18 px | 16 px | 400 | 1.55 |
| Body | 16 px | 15 px | 400 | 1.5 |
| Meta / caption | 14 px | 13 px | 500 | 1.4 |

**Требования:**

- На каждой странице ровно один элемент уровня Display / H1.
- Не допускается прыжок через уровень (H2 → H4). Если нужна визуальная иерархия без семантики — использовать `<div class="text-h3-like">`, а не ломать HTML-семантику.

### 1.2. Ритм секций и внешние отступы

| Контекст | Desktop | Mobile |
|---|---|---|
| Между крупными секциями (`<section>`) | 120 px | 64 px |
| H2 секции → подзаголовок | 16 px | 12 px |
| Подзаголовок секции → контент | 48 px | 32 px |
| Между карточками внутри секции | 24 px | 16 px |
| Horizontal padding контейнера | 64 px | 20 px |
| Max-width контейнера | 1200 px | 100% |

### 1.3. Шрифтовая пара

**Решение.** Один grotesque sans-serif с кириллической поддержкой и 4 начертаниями (400, 500, 600, 700). Варианты в порядке приоритета:

1. **Inter** — open-source, проверенная кириллица, де-факто стандарт для digital.
2. **Manrope** — более современный, немного легче по воздуху, кириллица сильная.
3. **Geologica** — open-source от Яндекса, кириллица-first, хорошо работает в заголовках.

**Один шрифт, одна гарнитура, без дополнительного serif для заголовков.** Усложнит поддержку, не даст явного выигрыша. Загрузка:

```css
@font-face {
  font-family: 'Inter';
  src: url('/fonts/Inter-Variable.woff2') format('woff2-variations');
  font-weight: 400 700;
  font-display: swap;
}
```

В `<head>` — `<link rel="preload">` для основных начертаний.

### 1.4. Эмодзи в UI — заменить на SVG-иконки

**Проблема.** Сайт активно использует эмодзи как иконки: `😫 Без платформы`, `😊 С платформой`, `📋`, `📞`, `😰`, `📝`, `🤷`, `📊`, `💻`, `🔴`, `🔔`, `💬`, `📅`, `🎉`, `💉`, `👨‍👩‍👦`, `⏱`, `🔒`, `📱`, `🤔`.

**Последствия:**

1. Разный рендеринг на iOS / Android / Windows — ломает tone of voice.
2. Screen readers читают эмодзи буквально — ломается accessibility.
3. Снижает воспринимаемое качество B2B-продукта.

**Решение.** Заменить все эмодзи в UI на SVG-иконки одного стиля:

- Библиотека: **Lucide** (open-source, 1000+ иконок, единая толщина линии) или **Phosphor**.
- Стиль: line-icons, толщина 1.5–2 px.
- Размер: 20×20 px для заголовков H3/H4, 16×16 px для inline body.
- Цвет: через CSS-токен `--icon-color` (по умолчанию наследует цвет текста).

**Маппинг эмодзи → Lucide:**

| Было | Стало |
|---|---|
| 😫 Без платформы | `frown` |
| 😊 С платформой | `smile` |
| 📋 Excel | `clipboard-list` |
| 📞 Обзвон | `phone` |
| 😰 Вопрос от мамы | `message-circle-warning` |
| 📝 Ручная запись | `pencil` |
| 🤷 Потерянная семья | `alert-circle` |
| 📊 Отчёт / дашборд | `bar-chart-3` |
| 💻 Панель | `laptop` |
| 🔴 Просрочено | `alert-circle` с токеном fill-red |
| 🔔 Уведомления | `bell` |
| 💬 Ответ | `message-circle` |
| 📅 Автозапись | `calendar-check` |
| 🎉 Добро пожаловать | `party-popper` |
| 💉 Вакцинация | `syringe` |
| 👨‍👩‍👦 Семьи | `users` |
| ⏱ Время | `clock` |
| 🔒 Защищено | `lock` |

| 🤔 FAQ | `help-circle` |

**Исключение.** `💜` в футере «Сделано с заботой» — можно оставить либо заменить на SVG-сердечко в фирменном цвете.

### 1.5. Иерархия заголовков (семантика H1–H6)

**Найденные нарушения:**

- **Главная.** Проверить, что метрики под Hero не помечены как h-элементы (должен быть div).
- **/for-clinics.** В секции «Что теряет ваша клиника сегодня» — прыжок H2 → H4, уровень H3 пропущен. Исправить: подзаголовки карточек сделать H3 либо div с классом `.text-h4-like` без семантики.
- **/for-families.** Проверить секции «Что даёт приложение» — подзаголовки экранов (`Ваш маршрут`, `Ваши назначения`, `Ваши документы`) должны быть H3.
- **/demo.** Каждая карточка роли — H2. Рассмотреть понижение до H3, оставив общий H2 «Выберите роль» в начале секции.

**Критерий приёмки.** Прогон через axe DevTools и headingsMap extension — 0 ошибок иерархии на каждой странице.

### 1.6. Межстрочные интервалы и длина строки

- **Body line-height:** 1.5 (не меньше).
- **Параграфы:** `margin-bottom: 1em`, не меньше.
- **Максимальная ширина строки текста:** 68–72 символа (≈ 640–680 px при 16–18 px). Сейчас в описательных блоках строки уходят на всю ширину секции — это мешает чтению. Ввести `max-width: 68ch` для `<p>` в описательных блоках.
- **Контрастность текста на фонах:** WCAG AA (4.5:1 для body, 3:1 для крупных заголовков). Проверка через WCAG Contrast Checker по всем комбинациям «текст + фон».

### 1.7. Вертикальный ритм карточек (ключевое для восприятия)

Внутри одной карточки:

```
┌──────────────────────────┐
│ [Title]                  │  ← H4, 18px 600
│   4px                    │
│ [Meta (время / статус)]  │  ← 14px 500, muted
│   12px                   │
│ [Description]            │  ← 15px 400
│   16px                   │
│ [Action button / badge]  │  ← либо inline, либо отдельно
└──────────────────────────┘
padding: 24px
```

Для timeline-элементов (маршрут, день координатора):

```
┌─────────────────────────────────┐
│ [Time] [Icon] [Title]           │  ← inline, gap 12px
│                4px              │
│         [Meta: подпись, длит.]  │  ← 14px, muted, offset слева под icon
│                12px             │
│         [Description]           │  ← 15px, offset слева
└─────────────────────────────────┘
padding между timeline-элементами: 24px
```

### 2.2. Блок «Что внутри маршрута» (замена сломанным «0+»)

Вместо четырёх нулевых счётчиков — четыре честные операционные метрики. Числа не про «достижения продукта», а про объём работы, который платформа берёт на себя:

| Метрика | Значение | Подпись |
|---|---|---|
| Длительность наблюдения | 104 недели | От первой явки до 2 лет ребёнка |
| Событий маршрута | 50+ | На каждую семью автоматически |
| Прививок под контролем | 18+ | По нацкалендарю РК |
| Напоминаний семье | ежедневно | Витамины, приёмы, анализы, прививки |

**Принцип.** Каждое число должно отвечать на вопрос «и что?» — то есть объяснять, почему клиника должна это заметить.

### 2.3. Блок «Знакомая ситуация?» на главной

Три карточки. Первые две работают как есть. Третью предлагаю переформулировать.

**Было.** «Руководитель не видит потерь. Большинство семей уходят после родов, но клиника узнаёт об этом слишком поздно.»

**Стало.**

> **Руководитель видит отчёт, когда уже поздно**
>
> Семья ушла в другую клинику три месяца назад. В квартальном отчёте это станет видно — если вообще станет.

### 2.4. Секция «Три интерфейса» на главной

**Проблема.** Декларируются три интерфейса, но в рендере читаемо только первый — интерфейс мамы. Табы «Приложение мамы / Панель координатора / Дашборд руководителя» визуально присутствуют, но переключение не работает либо не очевидно.

**Решение:**

1. Табы переключают контент инлайн без перезагрузки, анимация fade 200 ms.
2. Каждый таб показывает: (а) свой мокап, (б) 4 буллета про возможности, (в) CTA «Попробовать эту роль в демо» с корректным deeplink.
3. На мобильном три таба превращаются в три последовательных блока (без табов) — пользователь скроллит и видит все три.

**Предлагаемые тексты для двух других ролей:**

**Панель координатора.**

> Очередь задач на день: просрочено, срочно, обычно. Клик «Позвонить» — прямо из карточки. Клик «Записать» — в одно окно. История коммуникаций видна под каждой семьёй.
>
> - Приоритизированная очередь задач

## Фаза 3 — Демо-режим (1–1.5 недели)

### 3.1. Проверка текущего состояния (первая задача)

Команда проходит каждую из 4 ролей в incognito с чистой сессии, фиксирует результат в матрице «роль × экран × статус» (скринкаст + чек-лист).

| Роль | Что проверяем |
|---|---|
| Мама / Родитель | Открывается ли кабинет? Есть ли 2–3 демо-семьи на выбор (беременность на разных сроках + ребёнок 3 мес + ребёнок 14 мес)? Работают ли все экраны: маршрут / назначения / документы / прививки / профиль? |
| Координатор | Есть ли очередь задач с разными статусами (просрочено / сегодня / завтра)? Работает ли «Позвонить» (хотя бы как симуляция)? Видна ли карточка семьи с историей? |
| Врач | Расписание на сегодня? Карточка пациента с историей? Свободные слоты? |
| Руководитель | Дашборд с метриками? Переключение когорт? Таблицы KPI? |

По результатам — решение по каждому экрану: оставить / доделать / заменить симуляцией.

### 3.2. Обязательные улучшения UX (независимо от результатов проверки)

1. **Deeplink по ролям** (см. п. 0.3): `/demo?role=mom|coordinator|doctor|director` открывает нужный кабинет.
2. **Панель «Вы в демо-режиме»** — тонкая фиксированная полоска сверху внутри кабинетов:
   > Демо-режим · Данные тестовые · [Сменить роль ▾] · [Оставить заявку]
3. **Onboarding-тур при первом входе в роль** — 3–4 шага подсветки: «вот ваш маршрут», «вот уведомления», «вот документы», «попробуйте записаться». Библиотека: `driver.js` или `shepherd.js`.
4. **Быстрое переключение ролей** — dropdown в углу, без возврата на /demo.
5. **CTA внутри демо** — после просмотра (скролл до низа кабинета либо через 60 сек) появляется ненавязчивая плашка: «Понравилось? → Обсудить подключение». Dismissable, не агрессивная.
6. **Мобильная версия демо** — обязательна. Большинство руководителей клиник открывают сайт с телефона.

### 3.3. Улучшения по контенту демо

1. **Демо-семьи с именами.** Не «Семья А / Б / В», а «Айгерим, 28 лет, 16-я неделя беременности, второй ребёнок», «Данияр, 4 мес., плановая прививка пентавакцина #2», «Амина, 14 мес., плановый осмотр». Имена дают эмпатию и узнаваемость.
2. **Готовые сценарии.** В каждой роли — 3 пресета состояния: «Утро координатора», «Просрочки», «Новая семья». Клик по пресету — загружает соответствующий набор данных.
3. **Мотив Умай.** В кабинете мамы — тонкая иллюстрация или подпись с отсылкой к метафоре: например, всплывающее приветствие «Умай оберегает маршрут Айгерим». Не перегружать — одна-две точки на кабинет.

**Проблема.** На странице /demo в конце есть фраза: «Аналогичный кабинет доступен для второго родителя (папа, бабушка)». Сейчас это мелкий текст-сноска. На деле — важная фича, которую руководитель клиники может недооценить.

### 5.2. Accessibility (WCAG 2.1 AA)

- Все интерактивные элементы доступны с клавиатуры + visible focus state (не `outline: none` без замены)
- `alt` на всех информативных изображениях, `alt=""` на декоративных
- Иконки (Lucide/Phosphor) получают `aria-label` либо `aria-hidden="true"`, если дублируют соседний текст
- Skip-to-content ссылка в начале страницы
- Цветовой контраст — проверка WCAG Contrast Checker по всем комбинациям
- Формы: `<label>` для каждого инпута, `aria-describedby` для сообщений об ошибках
- Прогон Lighthouse Accessibility — цель 95+

### 5.3. Производительность (Core Web Vitals)

Целевые метрики:

- **LCP** < 2.5s
- **INP** < 200ms
- **CLS** < 0.1

Действия:

- Шрифты: `preload` основных начертаний + `font-display: swap`
- Все иллюстрации в `.webp` или `.avif` + responsive `srcset`
- Мокап телефона на главной — если SVG, оставить SVG; если растр — сжать и предоставить 2x/3x
- Lazy-load всего, что ниже первого экрана: `loading="lazy"` на `<img>`, IntersectionObserver для секций
- Убрать тяжёлые библиотеки анимации счётчиков (`react-countup` и аналоги) — см. п. 0.1. Заменить на CSS-only подход.
- Проверить CLS от отложенной загрузки шрифтов — резервировать высоту заголовков

### 5.4. PWA для маркетингового сайта

Сам сайт тоже стоит сделать PWA — для единообразия с приложением семей:

- `manifest.webmanifest`
- Service worker с кешированием статики
- Иконки 192, 512, 1024
- Даёт возможность «добавить на главный экран» для демо-режима
